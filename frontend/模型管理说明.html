模型管理平台（Model Management Platform, MMP）产品需求文档（PRD）

一、产品概述

1.1 产品目标

模型管理平台（MMP）是一个面向工程研发全生命周期的统一模型资产注册、管理与服务平台。其核心任务是对来自建模工具（如 MATLAB、Simulink、Modelica、SysML）中产生的模型进行集中管理，提供版本控制、权限控制、结构元数据存储、项目引用接口，避免模型碎片化、权限混乱和版本不可控。

1.2 产品作用定位

MMP 是一个“模型注册和服务中台”，不承担建模任务本身，也不负责显示项目结构，而是为以下外部系统提供服务：

  • ✅ 用户控制台系统：用于工程师管理自己的任务、上传模型、创建模型等

  • ✅ 项目管理系统：用于展示项目结构、任务分派、项目与模型的引用关系

MMP 通过权限机制控制用户对模型的访问，所有模型在物理存储、元数据管理、权限体系上都统一在 MMP 中完成。

二、核心功能模块

2.1 模型注册与版本管理

  • ⛳️ 支持模型上传、注册

    • 工程师通过用户控制台（可以是仿真工具或者统一的用户建模环境）上传模型文件（如 .slx、.mo、.sysml），这些文件将被传入模型管理平台（MMP），生成一个模型记录。

    • 把“模型”正式纳入 MMP 统一管理体系中，记录其基本信息、归属人、描述等。

  • ⛳️ 支持通过建模工具插件（如 MATLAB 插件）上传模型文件及接口说明文档

    • 在 MATLAB、Simulink 等建模工具中，可以通过插件（如按钮或命令）直接将模型上传到 MMP。如上传：

      • 模型文件（如 .slx、.mdl、.m）

      • 模型接口说明（如 JSON 格式的输入/输出/参数清单）

      • 可选的附属文档（说明文档、图片等）

  • ⛳️ 每个模型可注册多个版本，支持对比、回滚、状态锁定

    • 每个模型（如 DriveController）可以有多个版本（如 v1.0, v1.1, v2.0），每个版本都是独立存储并带有元数据。

    • 提供功能：

      • 版本比较（查看差异）

      • 版本回滚（将主版本退回到历史版本）

      • 版本锁定（冻结版本供项目引用，避免被误删/替换）

  • ⛳️ 模型必须通过评审后才能进入“已发布”状态供项目绑定

    • 上传之后模型的状态是 draft，必须经过评审流程（通常由架构师或项目负责人操作）并被标记为 published 后，才能绑定到项目。 避免未经审核的模型被误用于项目，确保模型质量合规。建议：

      • 模型评审发生在 MMP 内部；

      • 项目引用仅能绑定状态为 published 的模型版本；

      • 审核记录写入 model_review_log。

2.3 权限与可见性控制

  • 每个模型有 owner（上传者）与权限控制字段：

    • private：仅上传者可见

    • project_shared：通过项目引用授权

    • public：平台所有用户可见

  • 支持用户授权给特定人（如架构师）协助评审、修改

每个模型在系统中具有唯一的 owner（即上传者），平台为每个模型设置一组访问级别标识字段（如下）

2.3.1 private：仅上传者本人可见

  • 用途场景：工程师在个人空间构建/测试模型，尚未提交到项目。

  • 访问限制：

    • 仅 owner 可查询、编辑、提交评审。

    • 项目系统或他人不可引用该模型。

  • 权限逻辑：

    • 不可被其他用户或系统调用。

    • 可由 owner 在控制台中操作“转为共享”或“提交到项目”。

2.3.2 project_shared：通过项目引用授权可见

  • 用途场景：模型被项目引用后，项目成员需能访问、查看、下载。

  • 访问限制：

    • 仅项目系统中配置为引用该模型的成员可见。

    • 项目系统通过 project_model_ref 建立绑定。

  • 权限逻辑：

    • MMP 不主动显示“项目空间”，而通过接口校验“此用户是否为该项目成员”，从而授权访问

2.3.3 public：平台所有用户可见

  • 用途场景：通用库模型、标准组件、示范模型等。

  • 访问限制：

    • 所有平台登录用户都可查询和预览（只读）。

    • 不可修改，除非为管理员或拥有者。

  • 权限逻辑：

    • 允许项目直接引用，无需提交评审。

    • 通常用于构建标准模型库或行业规范组件。

2.4 模型引用接口

  • 提供外部项目系统进行“模型引用”操作（即建立 project → model_version 的关系）

  • 引用支持两种策略：固定版本引用、主版本自动升级引用

  • MMP 不维护项目结构，仅提供“某项目引用了哪些模型”接口

“模型引用”是指：

🍍 项目管理系统在其任务或结构中，引用某个已注册的模型版本，并将其作为该项目中的“使用模型”。

  • 引用 ≠ 复制模型；

  • 引用只建立逻辑关系；

  • 模型文件本体仍托管在 MMP 中。

🧩 功能 2：支持两种引用策略

  1. 固定版本引用（fixed）

“项目永远使用 v2.0 这个版本，无论模型后续是否更新。”

  • 使用场景：

    • 要求结果可复现、仿真稳定；

    • 敏感模型不希望自动更新。

  1. 主版本自动更新（latest）

“项目会自动跟随模型的最新发布版本（如 v2.2 → v2.3）。”

  • 使用场景：

    • 模型频繁优化、项目期望持续集成；

    • 需要始终使用最新优化结果。

MMP 只在引用中标记 bind_type，由项目系统决定是否定期拉取最新版本。

🧩 功能 3：MMP 不维护项目结构，只提供查询接口

MMP 不负责存储项目结构或任务树，仅提供：

  • 查看某个模型在哪些项目中被引用；

  • 给管理员分析模型的“使用情况”；

  • 支持项目对模型更新的追踪。

2.5 审批与生命周期管理

模型在被项目或团队使用之前，必须确保质量、结构完整、接口规范一致。因此，每个模型从上传到可被使用，必须经历一个标准的生命周期：

xxxxxxxxxx
草稿 → 待评审 → 已发布 → 项目绑定 → 历史归档

  • 模型提交后自动进入“待评审”状态

  • 审批流程：上传 → 审查 → 通过（进入注册表）或驳回（退回用户控制台）

  • 审查意见、流程记录入审计日志

✅ 模块1. 模型提交后进入 待评审 状态

  • 📌 触发条件：

    • 工程师通过控制台（如客户端工具，如matlab或者Dymola工具的插件或者云端工具）提交模型到 MMP；

    • 或从控制台的已有模型版本中“提交到项目”。

  • 📌 状态变化**：

    xxxxxxxxxx
    initial（草稿） → pending_review（待评审）

  • 📌 保存内容**：

    • 模型文件

    • 结构 JSON、接口元数据

    • 提交人、时间、用途说明

✅ 模块2. 审批流程说明

📌 标准流程

上传者提交模型
      ↓
MMP 进入评审队列
      ↓
评审人（架构师/项目负责人）在 MMP 中查看结构、接口、文档
      ↓
选择：通过 or 驳回

📘 评审结果：

  • 通过（Approved）：

    • 模型状态更新为 published

    • 可被项目管理系统引用

    • 显示在模型检索结果中（若权限允许）

  • 驳回（Rejected）：

    • 模型状态退回为 draft，并同步到用户控制台

    • 控制台显示“驳回原因”，支持重新上传修订版

🧩 关键问题

  1. 项目中是否需要存储模型文件？

不需要，项目管理系统只需记录模型引用信息。模型实际文件、版本、状态由 MMP 统一维护。

  1. 模型评审完成后，是否立即进入项目？

只有 “发布状态（Published）” 的模型才允许建立项目引用。评审中（Reviewing）模型不能被项目引用。

  1. 多个项目引用了同一个模型，怎么追踪？

MMP 中每个模型提供“被哪些项目引用”的查询接口，用于追溯依赖关系与影响分析。

 

 

四、流程示例：MATLAB 模型提交过程

📍 场景目标

某项目在项目管理系统中下发任务，要求工程师提交控制器模型。工程师使用 MATLAB 建模，并通过模型管理平台（MMP）提交模型。审核通过后，该模型版本被项目管理系统正式引用。

🧭 流程步骤（面向 MMP+控制台+项目系统的协同）

 

1. 项目任务下达（📍在 项目管理系统）

  • 项目管理员在项目管理系统创建任务： 控制器模型开发

  • 系统将该任务同步推送至工程师的 用户控制台

2. 用户控制台接收任务并响应

  • 工程师登录控制台，看到任务通知

  • 点击“提交模型”，系统提供三种方式：

    ✅ 上传本地 .slx 文件（MATLAB 模型） ✅ 启动 MATLAB 插件构建模型后上传 ✅ 从 MMP 中查找已有模型（已发布状态）发起引用申请

3. 模型上传至 MMP

  • 控制台统一调用 MMP 的模型注册接口

  • MMP 接收并存储以下内容：

    • 模型主文件（如 .slx）

    • 接口描述文档（如 .docx, .json）

    • 参数与结构元数据（解析后以结构化 JSON 存入数据库）

  • 模型状态初始设为：draft

4. 模型评审流程（MMP 内部流程）

  • 项目相关角色（如系统架构师）登录 MMP 平台

  • 评审动作包括：

    • 查看模型元数据、结构图、参数、版本对比

    • 给出评审意见

    • 审核通过/驳回

  • 结果处理：

    • ✅ 通过：模型状态设为 published，进入模型注册区

    • ❌ 驳回：模型状态仍为 draft，通知控制台用户修改重传

  • 所有操作自动写入审计日志（含操作人、时间、意见）

5. 控制台或管理员在项目管理系统发起“引用模型”操作：

xxxxxxxxxx
POST /api/project_model_ref
{
  "project_id": "P001",
  "model_id": "M123",
  "version": "v1.0.1",
  "bind_strategy": "locked"

五、数据库结构概要

5.1 模型管理数据库（MMP）

💡 模型管理平台的核心数据库，所有模型的注册、评审、版本、接口元数据、权限控制等都由此系统负责存储与调度。

主要有如下数据表组成：

 

1. model 表（模型主表）

说明：

  • 这是每个模型的“主档案”。

  • 与 model_version 是一对多关系（一个模型有多个版本）。

  • visibility 用于权限控制逻辑。

    示例：🧩 Step 1：上传模型（入库 model 表）

  1. model_version 表（模型版本信息）

说明：

  • 模型评审是对版本级别的，而不是模型主表。

  • status 会驱动评审流程控制。

  • 项目引用的是某一具体版本。

    示例：🧩 Step 2：上传版本（入库 model_version 表）

    张三上传了 v1.0 版本的模型文件：

3. model_metadata 表（元数据结构）

说明：

  • 这个字段是面向模型可组合、可查询、可验证设计的核心。

  • 上传后自动提取（如 MATLAB API 输出接口），供平台分析使用。

  • 支持与建模插件联动上传结构 JSON（或由系统解析）。

  • 用户个人上传的模型默认 private，需要授权给其他用户。

  • 若模型为 project_shared，平台会自动加入项目成员可查看的权限集。

  • 在模型状态变更时，必须记录操作人和动作。

  • 提高审计合规性与过程可追溯性。

5.2 项目系统数据库（PMS / 非 MMP 管理）

💡 项目系统独立管理项目、任务与引用模型记录，它不保存模型本体，只与 MMP 建立引用关系。

1. project 表

说明：

  • 与模型没有直接关系，但后续任务调度会涉及到模型开发需求。

     

2. project_task 表（项目任务清单）

说明：

  • 用于用户控制台任务视图展示。

  • 可以联动模型管理状态。

     

3. project_model_ref 表（项目与模型版本绑定）

说明：

  • 项目引用模型的核心表。

  • 通过调用 MMP API 创建此条绑定。

  • 若模型更新且为 auto_follow_main，引用会自动指向主版本。
